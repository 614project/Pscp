# PSCP 트랜스파일러 가이드 v0.4

## 목적

이 문서는 두 파트로 구성됩니다:

- **파트 I — 구현 가이드**
- **파트 II — 최적화 가이드**

이 문서의 목표는 실용적입니다:

1. PSCP 트랜스파일러 구현을 위한 권장 구조를 정의하고,
2. 구문(syntax)과 의미(semantics)를 어디서 해석해야 하는지 정의하며,
3. 범용 헬퍼를 거치지 않고 직접 하강(lowering)해야 하는 구문을 정의하고,
4. 참조 품질 백엔드에 기대되는 최적화를 정의합니다.

이 문서는 공식 언어 명세 및 API 명세를 보완합니다.

# 파트 I — 구현 가이드

## 1. 전체 목표

PSCP 트랜스파일러는 PSCP 소스를 유효하고, 가독성 높으며, 효율적인 C#으로 변환해야 합니다.

트랜스파일러는 다음을 수행해야 합니다:

- PSCP 소스의 의미(semantics)를 보존하고,
- 적절한 경우 C#/.NET의 패스스루(pass-through) 표면을 유지하며,
- PSCP 전용 문법 설탕(syntactic sugar)을 초기에 제거하고,
- 하강(lowering) 결정을 명시적이고 예측 가능하게 유지하며,
- 너무 많은 의미를 런타임 헬퍼에 위임하지 않도록 합니다.

---

## 2. 권장 파이프라인

실용적인 구현 파이프라인은 다음과 같습니다:

1. 렉싱(lexing),
2. 파싱(parsing),
3. 구문 보존 AST 구성,
4. 이름 바인딩,
5. 타입 및 형태 분석,
6. 의미론적 역설탕화(semantic desugaring),
7. 백엔드 지향 하강(lowering),
8. C# 코드 생성(emission),
9. 선택적 포맷팅.

권장 내부 단계:

```txt
TokenStream
SyntaxTree
AstTree
BoundTree
LoweredTree
CSharpEmitter
```

---

## 3. 필수 아키텍처 규칙

트랜스파일러는 다음을 명확히 분리해야 합니다:

- **구문(statement) 하강**
- **표현식(expression) 하강**

일반적인 구문 하강을 컴파일러가 생성한 람다나 표현식 청크(thunk)로 제어 흐름을 감싸서 해결하지 마십시오.

올바른 대안이 없는 경우를 제외하고, 다음 구문들은 반드시 구문(statement) 수준으로 하강해야 합니다:

- `if` 문,
- `while` 문,
- `for` 문,
- `break`,
- `continue`,
- `return`,
- 선언 기반 입력 축약 문법,
- 구문 기반 출력 축약 문법.

## 5. 프론트엔드의 책임

하강이 시작되기 전에, 프론트엔드는 최소한 다음 범주들의 차이를 이미 파악하고 있어야 합니다.

### 5.1 선언(Declarations)

다음을 구분해야 합니다:

- 불변(immutable) 선언,
- 가변(mutable) 선언,
- 추론된(inferred) 선언,
- 명시적 타입 선언,
- 선언 기반 입력 축약 문법,
- 크기 지정 배열 선언,
- 알려진 컬렉션 자동 생성 선언,
- 대상 타입 할당 축약 선언.

### 5.2 출력 구문(Output Statements)

다음을 구분해야 합니다:

- `= expr` 쓰기 구문,
- `+= expr` 줄바꿈 포함 쓰기 구문,
- 일반적인 `+=` 대입,
- 알려진 자료구조 연산자 재작성 후보.

### 5.3 호출(Calls)

다음을 구분해야 합니다:

- 일반적인 괄호 호출,
- 공백 구분 호출,
- 수정된 인수(`ref`, `out`, `in`),
- out 변수 선언,
- `out _` 버리기(discard),
- 내장 집계(aggregate) 호출,
- 변환 키워드 호출.

### 5.4 컬렉션 및 제너레이터(Collections and Generators)

다음을 구분해야 합니다:

- 구체화된(materialized) 컬렉션 표현식 `[...]`,
- 구체화된 빌더,
- `for` 또는 `->`가 있는 제너레이터 표현식 `(...)`,
- 범위 요소,
- 스프레드 요소.

### 5.5 대입 형태(Assignment Forms)

다음을 구분해야 합니다:

- 일반적인 대입 구문,
- 복합 대입,
- 명시적 대입 표현식 `:=`.

### 5.6 타입 본문 축약(Type-Body Shorthand)

다음을 구분해야 합니다:

- 패스스루 접근 제한자(`public`, `private`, ...),
- 접근 섹션 레이블(`public:`, `private:`, ...),
- `operator<=>(other)` 축약 문법,
- 일반적인 패스스루 연산자/멤버 선언.

## 7. 엄격하게 적용해야 하는 파싱 규칙

### 7.1 `public` vs `public:`

이 둘은 문맥상 서로 다른 토큰이며, 혼동해서는 안 됩니다.

- `public`  = 패스스루 C# 접근 제한자
- `public:` = PSCP 섹션 레이블

동일한 규칙이 다음에도 적용됩니다:

- `private`
- `private:`
- `protected`
- `protected:`
- `internal`
- `internal:`

### 7.2 괄호 안에서 제너레이터 감지

괄호로 감싸진 형태가 `GeneratorExpr`로 처리되려면, 직접 내포된 형태에 `for` 또는 `->`를 사용하는 인식된 반복자 구문이 포함되어야 합니다.

제너레이터로 파싱해야 하는 예:

```txt
(0..<n -> i do i * i)
(0..<n -> i { i * i })
(for i in 0..<n do a[i])
```

일반 괄호 표현식으로 남아야 하는 예:

```txt
(1 + 2)
(x)
(a, b)
(0..<n)
```

### 7.3 출력 축약 문법은 구문의 시작에서만

`= expr`과 `+= expr`은 구문의 시작 위치에서만 출력 구문으로 처리됩니다.

### 7.4 `:=`

`:=`는 값을 반환하는 독립적인 대입 연산자로 파싱해야 하며, 일반 대입과 통합해서는 안 됩니다.

### 7.5 알려진 자료구조 연산자 후보

다음과 같은 표현식들:

```txt
visited += me
--stack
~queue
```

은 먼저 일반 연산자로 파싱해야 합니다. 알려진 자료구조 재작성은 이후 수신자(receiver) 타입을 기반으로 바인딩/하강 단계에서 수행됩니다.

## 8. 바인딩 및 의미 분석

바인더(binder)는 다음을 해석해야 합니다:

- 심볼 정체성,
- 가변성(mutability),
- 재귀 허용 여부,
- 표현식 타입,
- 알려진 컬렉션 자동 생성 적용 가능 여부,
- 알려진 자료구조 연산자 적용 가능 여부,
- 집계 내장 함수의 오버로드 패밀리 선택,
- 변환 키워드 의도,
- `T.asc` / `T.desc`에 대한 순서 제공자 가용성,
- 메서드에 대한 섹션 레이블 의도,
- 블록 값 동작.

---

## 9. 버리기(Discard) 의미론

PSCP의 `_`는 항상 의미론적 버리기(discard)입니다.

백엔드 규칙:

- C#이 지원하는 경우 진정한 C# discard를 생성하고,
- 그렇지 않으면 `__pscpDiscard0`과 같은 사용되지 않는 이름을 생성하며,
- 소스의 `_`가 생성된 백엔드 의미론에서 의도치 않게 의미 있는 변수가 되는 일이 없도록 합니다.

이 규칙은 특히 다음 경우에 중요합니다:

- 람다 매개변수,
- 반복 변수,
- 분해(deconstruction) 바인딩,
- 하강 중 도입된 합성 지역 변수.

## 11. 입력 구문 하강

### 11.1 스칼라 입력

```txt
int n =
```

권장 하강 형태:

```csharp
int n = __pscpIn.Int();
```

### 11.2 다중 스칼라

```txt
int n, m =
```

권장 하강 형태:

```csharp
int n = __pscpIn.Int();
int m = __pscpIn.Int();
```

### 11.3 고정 길이 배열

```txt
int[n] arr =
```

권장 하강 형태:

```csharp
int[] arr = new int[n];
for (int __pscpIter0 = 0; __pscpIter0 < n; __pscpIter0++)
    arr[__pscpIter0] = __pscpIn.Int();
```

### 11.4 튜플 배열

```txt
(int, int)[m] edges =
```

권장 하강 형태:

```csharp
(int, int)[] edges = new (int, int)[m];
for (int __pscpIter0 = 0; __pscpIter0 < m; __pscpIter0++)
    edges[__pscpIter0] = (__pscpIn.Int(), __pscpIn.Int());
```

참조 백엔드는 이러한 핫 경로(hot path)에서 리플렉션 방식의 범용 읽기 헬퍼를 사용하는 것을 피해야 합니다.

## 12. 출력 구문 하강

### 12.1 `= expr`

줄바꿈 없이 쓰는 연산으로 하강합니다.

### 12.2 `+= expr`

줄바꿈 포함 쓰기 연산으로 하강합니다.

## 13. 구체화된 컬렉션 하강

### 13.1 `[]`는 구체적인 결과를 의미

구체화된 컬렉션 표현식은 대상 문맥에 따라 배열, 리스트, 연결 리스트 등 구체적인 구성 방식으로 하강해야 합니다.

### 13.2 배열 대상 문맥

직접 할당 및 채우기를 선호합니다.

### 13.3 알려진 컬렉션 대상

직접 컬렉션 생성 및 추가를 선호합니다.

### 13.4 구체화된 범위 사용

직접 채우기가 가능한 경우, `[]` 안의 범위는 범용 런타임 범위 헬퍼로 남기지 않아야 합니다.

---

## 14. 제너레이터 표현식 하강

### 14.1 원칙

제너레이터 표현식은 이터러블을 생성하는 형태이며, 구체화를 의미하지 않습니다.

### 14.2 집계 소비자

`sum`, `min`, `max`, `sumBy`, `minBy`, `maxBy`와 같은 집계 내장 함수가 소비하는 경우, 직접 인라인이 가능하다면 참조 백엔드는 중간 이터레이터 객체를 생성하는 대신 직접 루프로 하강하는 것을 선호해야 합니다.

### 14.3 일반 이터러블 문맥

더 넓은 이터러블 문맥에서 사용되는 경우에는 헬퍼 열거자 또는 생성된 이터레이터를 사용하는 것도 허용됩니다.

---

## 15. 범위(Range) 하강

### 15.1 문맥 의존적 정책

범위 하강은 문맥에 따라 달라집니다.

### 15.2 순수 루프 문맥

단순 숫자 루프의 경우:

```txt
for i in 0..<n { ... }
0..<n -> i { ... }
```

직접 C# `for` 루프를 선호합니다.

### 15.3 구체화된 빌더 문맥

다음과 같은 경우:

```txt
[0..<n -> i do f(i)]
```

직접 할당 및 직접 채우기를 선호합니다.

### 15.4 집계 제너레이터 문맥

다음과 같은 경우:

```txt
sum (0..<n -> i do f(i))
```

직접 루프 누적을 선호합니다.

### 15.5 단독 이터러블 문맥

더 직접적인 하강이 적합하지 않은 경우, 헬퍼 이터러블을 사용하는 것도 허용됩니다.

---

## 16. 빠른 반복 `->` 하강

### 16.1 핵심 규칙

참조 백엔드는 직접 `for` 하강이 가능한 경우, 단순 숫자 범위 반복을 헬퍼 기반의 `foreach`로 하강해서는 안 됩니다.

이 규칙은 다음에 적용됩니다:

- 숫자 범위에 대한 구문 형태의 빠른 반복,
- 숫자 범위에 대한 인덱스 기반 빠른 반복,
- 범위 기반 빌더 하강,
- 범위 기반 집계 제너레이터 하강.

### 16.2 권장 처리 방식

- 범위 + 인덱스 바인딩 -> 직접 `for`,
- 임의 접근 컬렉션 + 인덱스 요청 -> 이점이 있는 경우 직접 인덱스 기반 `for`,
- 그 외 -> `foreach`를 사용하는 것도 허용됩니다.

---

## 17. 집계 내장 함수 패밀리 하강

집계 내장 함수는 소스/API 수준에서는 일반 호출로 하강해야 하지만, 참조 백엔드에서는 직접 루프 또는 비교 트리로 처리해야 합니다.

### 17.1 이진(binary) 및 가변 인수 `min` / `max`

- 소수의 고정 인수 -> 직접 비교 트리,
- 기본 숫자 타입 -> 직접 비교 또는 `Math.Min`/`Math.Max` 동등 처리,
- 순서가 있는 비기본 타입 -> 비교자 기반 비교 트리.

### 17.2 이터러블 `min` / `max`

- 명시적 반복,
- 초기화 상태 추적,
- 현재 최적값 갱신.

### 17.3 `sum`

- 명시적 반복,
- 직접 누적.

### 17.4 `sumBy`

- 명시적 반복,
- 요소마다 선택자(selector) 한 번 평가,
- 선택자 결과를 누적.

### 17.5 `minBy` / `maxBy`

- 명시적 반복,
- 요소마다 키(key) 한 번 평가,
- 현재 최적 요소 및 키 추적.

### 17.6 `chmin` / `chmax`

가능한 경우 직접 조건부 재작성을 선호합니다.

권장 백엔드 스타일:

```csharp
if (value < target) { target = value; return true; }
return false;
```

또는 `chmax`에 해당하는 동등 형태.

## 18. 변환 키워드 호출 하강

### 18.1 원칙

변환 키워드 호출은 의미론적으로 분류한 후, 소스 타입과 대상 타입 범주에 따라 하강해야 합니다.

### 18.2 범주별 권장 전략

#### 18.2.1 문자열 -> 숫자

파싱 호출로 하강합니다.

#### 18.2.2 숫자 -> 숫자

직접 캐스트/변환 표현식으로 하강합니다.

#### 18.2.3 불리언 -> 숫자

조건부 `0/1` 또는 동등한 직접 변환으로 하강합니다.

#### 18.2.4 숫자 -> 불리언

`value != 0` 스타일의 의미론으로 하강합니다.

#### 18.2.5 문자열 -> 불리언

언어/API 계약에 따라, 빈 문자열/비어 있지 않은 문자열 검사 등 광범위한 진리값(truthiness) 논리로 하강합니다.

참조 백엔드는 이러한 핫 스칼라 변환을 리플렉션 기반의 범용 방식으로 처리하는 것을 피해야 합니다.

## 19. `:=` 하강

### 19.1 의미

`lhs := rhs`는 값을 반환하는 대입입니다.

### 19.2 하강 전략

C#이 주변 표현식 문맥에서 대입을 자연스럽게 표현할 수 있다면, 직접 대입 표현식을 생성합니다.

그렇지 않으면, 다음을 보존하는 방식으로 임시 변수를 도입하여 하강합니다:

1. 부수 효과(side-effect) 순서,
2. 대입된 값의 결과,
3. 가능한 경우 가독성.

---

## 20. 알려진 자료구조 연산자 재작성 하강

이러한 재작성은 백엔드에 통합되어야 하며, 범용 런타임 래퍼로 구현해서는 안 됩니다.

### 20.1 `HashSet<T>`

- `set += x` -> `set.Add(x)`
- `set -= x` -> `set.Remove(x)`

반환되는 `bool` 값은 반드시 보존해야 합니다.

### 20.2 `Stack<T>`

- `s += x` -> `s.Push(x)`
- `~s` -> `s.Peek()`
- `--s` -> `s.Pop()`

### 20.3 `Queue<T>`

- `q += x` -> `q.Enqueue(x)`
- `~q` -> `q.Peek()`
- `--q` -> `q.Dequeue()`

### 20.4 `PriorityQueue<TElement, TPriority>`

- `pq += (item, priority)` -> enqueue,
- `~pq` -> peek으로 반환된 item,
- `--pq` -> dequeue로 반환된 item.

우선순위가 필요한 경우, `TryPeek` / `TryDequeue`와 같은 일반적인 명시적 API는 그대로 유지되어야 합니다.

## 21. `new[n]` 및 `new![n]` 하강

### 21.1 `new[n]`

문맥에서 대상 타입을 해석한 후, 올바른 요소 타입의 배열을 할당합니다.

### 21.2 `new![n]`

문맥에서 대상 요소 타입을 해석합니다.

그 후:

1. 배열을 할당하고,
2. 알려진 컬렉션 규칙에 따라 암시적 `new()`를 사용하여 각 요소를 초기화합니다.

요소 타입이 지원되는 알려진 컬렉션 집합에 없는 경우, 이 재작성은 거부해야 합니다.

## 22. 접근 섹션 레이블 하강

### 22.1 기능의 성격

접근 섹션 레이블은 소스 수준의 의도 표현자이며, 완전히 강제되는 백엔드 접근 제어 선언이 아닙니다.

### 22.2 필수 동작

트랜스파일러는 다음을 수행해야 합니다:

- `public:`, `private:`, `protected:`, `internal:` 레이블을 인식하고,
- 명시적 접근 제한자가 없는 이후 메서드에 대한 의도된 접근성을 추적하며,
- 선택적으로 경고를 발생시키고 접근 위반 검사를 수행합니다.

### 22.3 비차단 정책

접근 섹션 레이블은 트랜스파일을 차단해서는 안 됩니다.

백엔드는 여전히 메서드를 `public`으로 생성할 수 있습니다.

이 정책은 콜론이 붙은 섹션 레이블에만 적용됩니다.
콜론 없이 사용된 일반 패스스루 접근 제한자는 이 경량 섹션 레이블 동작의 대상이 아닙니다.

## 23. 타입 본문에서 순서 축약 문법 하강

`operator<=>(other)` 축약 문법은 해당 타입의 기본 순서를 보존하는 형태로 하강해야 합니다.

가능한 백엔드 전략으로는 다음이 있습니다:

1. `CompareTo` 메서드를 생성하거나 연결하기,
2. 비교자 축약 문법에서 사용하는 비교 헬퍼를 생성하기,
3. 백엔드 스타일에 따라 직접 비교 멤버를 생성하기.

이 축약 문법은 단순한 임의 헬퍼 메서드가 아니라, 소스 수준의 순서 정의로 동작해야 합니다.

트랜스파일러는 가능한 경우 C#/.NET의 패스스루 표면을 보존해야 합니다.

# 파트 II — 최적화 가이드

## 25. 최적화 철학

PSCP 트랜스파일러의 최적화 목표는 추상적인 전체 프로그램 수준의 영리한 최적화가 아닙니다.

대신, 참조 백엔드는 매우 구체적인 스타일의 최적화에 집중해야 합니다:

1. 경쟁 프로그래밍에 특화된 일반적인 소스 형태를 조기에 감지하고,
2. 불필요한 런타임 추상화 생성을 피하며,
3. 고수준 PSCP 문법 설탕을 직접적인 C# 제어 흐름 및 저장 연산으로 변환하고,
4. 생성된 코드의 가독성과 디버깅 용이성을 유지합니다.

PSCP에서 가장 좋은 최적화는 종종 초기의 구조적 결정입니다:

- 의미론적으로 할당이 필요하지 않으면 할당하지 않고,
- 직접 인덱싱이 가능하면 열거하지 않으며,
- 구문을 직접 생성할 수 있으면 람다로 감싸지 않고,
- 고정된 의미론을 리플렉션, `dynamic`, 또는 범용 헬퍼로 미루지 않습니다.

즉, PSCP 최적화는 근본적으로 **역설탕화 규율과 형태 기반 하강**이며, 후반부의 영리한 최적화가 아닙니다.

## 26. 최적화 합법성 규칙

모든 최적화는 다음을 보존해야 합니다:

1. 소스 평가 순서,
2. 소스 부수 효과,
3. 변경 타이밍,
4. 관찰 가능하고 관련 있는 경우의 예외 동작,
5. 암시적 vs 명시적 구체화 의도,
6. API 명세에 정의된 정확한 진리값/변환 의미론,
7. `:=`의 값 반환 동작,
8. 알려진 자료구조 연산자 재작성의 반환값 보존.

최적화는 다음을 절대 암묵적으로 변경해서는 안 됩니다:

- 컬렉션의 구체화 여부,
- 제너레이터의 탈출 여부,
- 연산의 값 반환 여부,
- 다른 부수 효과 전후의 대상 변경 여부.

## 29. 피해야 할 필수 안티패턴

다음은 참조 백엔드에서 금지되거나 강력히 권장하지 않는 패턴입니다.

### 29.1 단순 범위에서 헬퍼 열거자 사용

직접 `for` 루프가 가능한 경우, 다음을:

```txt
0..<n -> i { ... }
```

다음과 같이 하강하지 마십시오:

```csharp
foreach (var i in SomeRangeHelper(...)) { ... }
```

### 29.2 핫 경로에서 리플렉션 방식의 스칼라 변환

기본 스칼라 입력 하강이나 일반적인 변환 키워드 하강에서 `Convert.ChangeType` 방식의 범용 전략에 의존하지 마십시오.

### 29.3 핵심 숫자 집계에서 `dynamic` 사용

참조 백엔드에서 `sum`, `sumBy`, `min`, `max`, 비교 헬퍼, 비교-갱신 헬퍼를 `dynamic`으로 구현하지 마십시오.

### 29.4 일반 구문에서 컴파일러 생성 람다 청크 사용

단순히 표현식 형태를 강제하기 위해 컴파일러 생성 람다로 구문 제어 흐름을 인코딩하지 마십시오.

### 29.5 제너레이터 기반 집계의 강제 구체화

기본적으로 다음을:

```txt
sum (0..<n -> i do f(i))
```

다음과 같이 하강하지 마십시오:

```csharp
ToArray(...).Sum()
```

### 29.6 단순 형태에서 무분별한 LINQ 하강

다음에 대해 기본으로 LINQ를 사용하지 마십시오:

- 형태 지정 입력,
- 단순 빌더,
- 일반적인 집계 호출,
- 범위 루프,
- `new![n]` 초기화,
- 직접 `HashSet` / `Queue` / `Stack` 재작성.

### 29.7 알려진 자료구조 연산자의 반환값 손실

소스 표현식이 불리언 문맥에서 사용될 때 `HashSet<T>.Add` / `Remove`의 불리언 결과를 버리지 마십시오.

## 31. 빠른 입력 최적화 가이드

### 31.1 핵심 규칙

스칼라 및 형태 지정 입력은 특수화된 스캐너 호출을 사용해야 합니다.

#### 31.1.1 권장 방식

```csharp
int n = __pscpIn.Int();
```

#### 31.1.2 지양해야 할 기본 방식

```csharp
int n = Convert.ToInt32(ReadGeneric());
```

### 31.2 특수화된 리더 집합

참조 백엔드는 최소한 다음 스캐너 진입점을 노출해야 합니다:

- `Int()`
- `Long()`
- `Double()`
- `Decimal()`
- `StringToken()`
- `CharToken()`
- `BoolToken()` 또는 동등한 것.

### 31.3 고정 길이 배열 읽기

권장 방식:

1. 한 번 할당,
2. 직접 루프로 채우기,
3. 요소마다 특수화된 스캐너 호출.

지양해야 할 방식:

- 기본 형태 지정 입력 하강에서 `Enumerable.Range(...).Select(...).ToArray()` 사용.

### 31.4 튜플 읽기

튜플 배열은 단일 명시적 루프로 하강해야 합니다.

튜플의 인수 개수가 알려져 있고 적다면, 필드별 읽기를 직접 인라인합니다.

### 31.5 격자(Grid) 읽기

중첩된 범용 열거자보다 중첩 루프 또는 특수화된 격자 헬퍼를 선호합니다.

### 31.6 줄 기반 리더

`stdin.line()`, `stdin.lines(n)`, `stdin.words()`, `stdin.chars()`는 줄 기반 헬퍼를 사용할 수 있지만, 핫 토큰 기반 스칼라 읽기가 기본으로 반복적인 줄 분할로 저하되어서는 안 됩니다.

### 31.7 입력 일괄 처리

백엔드가 스캐너 추상화를 사용하는 경우, 버퍼 상태를 캐시하고 하위 수준 스캔이 가능할 때 토큰별 문자열 분할을 피해야 합니다.

### 32.2 스칼라 쓰기

### 32.4 구조화된 격자(Grid) 출력

### 32.6 보간 문자열 최적화

### 32.7 타이트한 루프 내 출력

핫 루프 내에서 반복적인 `+= expr`을 하강할 때, 최소한의 헬퍼 오버헤드로 직접 writer 호출을 선호합니다.

## 33. 범위 최적화 가이드

### 33.1 직접 숫자 범위 루프 조기 감지

범위가 다음 조건을 만족하는 경우:

- 정수/long 기반,
- 단조(monotonic),
- 루프 문맥에서 사용,

직접 `for` 하강을 선호합니다.

#### 33.1.1 포함/미포함 처리

포함/미포함 선택은 단순한 핫 케이스에서 런타임 불리언 플래그로 전달하는 대신 생성된 루프 조건에 인코딩해야 합니다.

#### 33.1.2 스텝 범위

스텝 방향과 종료 조건이 정적으로 명확한 경우, 스텝 범위도 직접 루프를 선호해야 합니다.

#### 33.1.3 상수 스텝 빠른 경로

스텝이 컴파일 타임 상수이고 0이 아닌 경우, 즉시 직접 루프를 생성합니다.

#### 33.1.4 내림차순 범위 빠른 경로

`n..-1..0`과 같은 내림차순 범위는 가능한 경우 내림차순 `for` 루프로 직접 하강해야 합니다.

### 33.2 빌더 문맥에서의 범위

정적으로 크기가 알려진 범위에 대한 구체화된 빌더는 가능한 경우 최종 크기를 계산하고 직접 채웁니다.

### 33.3 제너레이터 집계 문맥에서의 범위

범위에 대한 제너레이터 기반 집계는 중간 할당 없이 단일 누적 루프로 하강해야 합니다.

### 33.4 탈출 문맥에서의 범위 값

범위 값이 일반 이터러블 문맥으로 탈출하는 경우 헬퍼 이터러블을 사용하는 것도 허용되지만, 핫 참조 케이스에서는 불필요한 플래그가 많은 시그니처를 피해야 합니다.

## 34. 구체화된 빌더 최적화 가이드

### 34.1 정적으로 크기가 알려진 빌더

다음의 경우:

```txt
[0..<n -> i do f(i)]
```

다음을 선호합니다:

1. 크기 `n` 계산,
2. 배열 한 번 할당,
3. `for` 루프에서 직접 채우기.

### 34.2 정적으로 크기가 알려진 블록 빌더

다음의 경우:

```txt
[0..<n -> i {
    let x = g(i)
    x + 1
}]
```

여전히 직접 할당 및 직접 채우기를 선호합니다.

### 34.3 크기를 정적으로 알 수 없는 빌더

최종 개수를 저렴하게 알 수 없는 경우, 필요에 따라 크기 조정 가능한 빌더와 최종 변환을 사용합니다.

### 34.4 스프레드 최적화

`[1, 2, ..xs, 6]`을 구체화할 때, 가능한 경우 총 길이를 사전 계산합니다. 가능하지 않은 경우 크기 조정 가능한 빌더로 순서대로 추가합니다.

### 34.5 빌더 형태 분류

유용한 분류 방식은 다음과 같습니다:

- **고정 크기**: 직접 배열 채우기,
- **저렴하게 계산 가능**: 리스트 사전 크기 지정 후 추가,
- **크기 불명**: 늘어나는 빌더 후 최종화.

### 34.6 탈출 인식 구체화

구체화된 빌더 결과가 즉시 인라인 가능한 문맥에서 소비되는 경우, 구체화가 의미론적으로 관찰 가능하지 않을 때에만 선택적으로 계산을 융합할 수 있습니다.

기본적으로 `[]`의 구체화 의도는 보존합니다.

## 35. 제너레이터 최적화 가이드

### 35.1 제너레이터 + 집계 융합

제너레이터 표현식이 집계 내장 함수를 직접 소비하는 경우, 루프 융합을 선호합니다.

예:

```txt
sum (0..<n -> i do f(i))
min (xs -> x do g(x))
sumBy (0..<n -> i do node(i)) h
```

권장 하강:

- 제너레이터 객체 없음,
- 중간 배열 없음,
- 중간 리스트 없음,
- 직접 루프 하나.

### 35.2 비집계 문맥에서의 제너레이터

제너레이터가 일반 이터러블 문맥으로 탈출하는 경우, 헬퍼 이터레이터를 사용하는 것도 허용됩니다.

### 35.3 제너레이터 + `minBy` / `maxBy`

제너레이터가 `minBy` 또는 `maxBy`를 소비하는 경우, 제너레이터 생성과 최적 요소 추적을 하나의 루프로 융합합니다.

### 35.4 제너레이터 + `count` / `any` / `all`

이것들이 집계 패밀리 내장 함수로 추가되거나 취급되는 경우, 제너레이터 기반 사용도 가능한 경우 조기 종료를 포함한 직접 루프로 융합해야 합니다.

### 35.5 중첩 제너레이터 방지

두 계층 모두 중첩 루프로 평탄화할 수 있는 경우, 중첩된 제너레이터를 계층화된 열거자 객체로 하강하는 것을 피합니다.

## 36. 집계 최적화 가이드

### 36.1 소수 고정 인수에 대한 `min` / `max`

다음의 경우:

```txt
max a b c d
```

중첩된 직접 비교 트리를 선호합니다.

임시 배열이나 리스트를 생성하지 않습니다.

### 36.2 `sum`

구체적인 누적 타입을 가진 직접 누적을 선호합니다.

#### 36.2.1 타입 특수화된 합계

피연산자 타입이 알려진 경우, 강하게 타입된 누적 변수를 사용하고 범용/동적 산술을 피합니다.

### 36.3 `sumBy`

요소마다 선택자를 한 번 평가하고, 직접 누적하며, 참조 백엔드에서 `Select(...).Sum()`을 피합니다.

### 36.4 `minBy` / `maxBy`

최적 요소와 최적 키를 동시에 추적합니다.
같은 요소에 대해 키를 재계산하지 않습니다.

### 36.5 `chmin` / `chmax`

대상이 단순한 백엔드 lvalue인 경우, 직접 비교-갱신 코드로 하강해야 합니다.

이것이 런타임 헬퍼 호출보다 더 빠르고 명확합니다.

### 36.6 빈 값 검사 전략

저렴한 count/length를 가진 컨테이너에 대한 이터러블 집계의 경우, 루프 진입 전 빠른 빈 값 검사가 이점이 있을 수 있습니다.

### 36.7 조기 종료 집계

향후 집계 내장 함수에 `any`, `all`, 또는 조건부 `count`가 포함되는 경우, 의미론이 허용하는 경우 단락 평가 또는 조기 종료 하강을 선호합니다.

## 37. 대입 표현식 최적화 가이드

### 37.1 가능한 경우 백엔드 네이티브 형태 사용

`:=`에 대해, 의미론을 보존하는 경우 백엔드의 네이티브 대입 표현식 기능을 사용합니다.

### 37.2 필요할 때만 임시 변수 도입

더 복잡한 경우에 평가 순서를 보존하기 위해 필요한 경우에만 임시 변수를 도입합니다.

### 37.3 전형적인 DSU 패턴

다음의 경우:

```txt
parent[x] := find(parent[x])
```

백엔드가 지원하는 가장 직접적이고 가독성 높은 대입 표현식을 생성합니다.

### 37.4 중첩 대입 표현식

다음과 같은 체인의 경우:

```txt
a = b := c
```

불필요한 임시 변수를 최소화하면서 오른쪽 결합(right-associative) 평가를 보존합니다.

### 37.5 부수 효과 안전성

왼쪽 피연산자에 부수 효과가 있거나 비싼 인덱싱/속성 접근이 반복되는 경우, 최종 대입 생성 전에 부분 표현식을 한 번만 호이스팅하는 것을 고려합니다.

## 38. 변환 최적화 가이드

### 38.1 파싱 vs 변환

문자열 파싱의 경우, 직접 파싱 호출로 하강합니다.

스칼라 간 변환의 경우, 캐스트 또는 직접 조건부 재작성을 선호합니다.

### 38.2 불리언 변환 단축

예:

- `int(boolExpr)` -> 삼항 연산자 또는 백엔드 숫자-불리언 변환,
- `bool(numExpr)` -> `numExpr != 0`,
- `bool(strExpr)` -> 빈/비어 있지 않은 문자열 검사.

### 38.3 핫 스칼라 케이스에서 범용 변환 헬퍼 지양

범용 변환 헬퍼는 존재할 수 있지만, 일반적인 스칼라 케이스에 대한 기본 참조 하강이 되어서는 안 됩니다.

### 38.4 상수 변환 폴딩

변환 키워드 호출의 소스가 컴파일 타임 상수이고 변환을 정적으로 평가하는 것이 안전한 경우, 상수 폴딩을 수행합니다.

예:

- `int "123"` -> 상수 `123`,
- `bool ""` -> 상수 `false`,
- `int true` -> 상수 `1`.

### 38.5 불필요한 변환 제거

소스 의미론이 변경되지 않는 경우, 다음과 같이 명백히 불필요한 변환을 제거합니다:

- 타입이 이미 일치할 때 `int(intExpr)`,
- 타입이 이미 일치할 때 `bool(boolExpr)`.

## 39. 알려진 자료구조 연산자 최적화 가이드

visited += x

## 40. `new![n]` 최적화 가이드

### 40.1 권장 하강

다음의 경우:

```txt
List<int>[] graph = new![n]
```

다음을 선호합니다:

1. 배열 할당,
2. 각 슬롯을 `new()`로 채우는 직접 루프 생성.

예시 형태:

```csharp
var graph = new List<int>[n];
for (int i = 0; i < n; i++) graph[i] = new();
```

### 40.2 제약 조건 확인

요소 타입이 컴파일러가 인식하는 자동 생성 가능한 타입이 아닌 경우 조기에 거부합니다.

### 40.3 루프 강도 감소

`new![n]`의 초기화 루프는 단순하고 핫하므로, 평탄하고 직접적으로 유지합니다.

`Enumerable.Range(...).Select(...)` 또는 유사한 방식을 통해 하강하지 마십시오.

## 41. 비교자 최적화 가이드

### 41.1 `T.asc` / `T.desc`

이것들은 의미론적으로 한 번 해석되고, 효율적인 비교자 객체 또는 직접적인 백엔드 구문으로 하강해야 합니다.

### 41.2 캐싱

의미론을 복잡하게 만들지 않는 경우, 비교자 인스턴스를 캐싱하는 것이 좋은 최적화입니다.

### 41.3 타입 본문 순서 축약

`operator<=>(other)`는 집계 비교 및 비교자 축약 문법에 사용되는 동일한 순서 시스템과 연결되어야 합니다.

### 41.4 비교 트리 단순화

기본 순서 타입에 대한 고정 인수 `min` / `max`의 경우, 비교자 디스패치 대신 직접 비교 트리를 선호합니다.

### 41.5 반복 비교 위치 재사용

같은 비교자 값이 지역 범위에서 여러 번 사용되는 경우, 이익이 되는 경우 하강된 비교자 객체를 재사용합니다.

---

## 42. 슬라이싱 및 인덱싱 최적화 가이드

### 42.1 가능한 경우 백엔드 네이티브 슬라이싱 보존

문자열과 1차원 배열에 대해, 효율적이고 가독성 높은 경우 백엔드 네이티브 슬라이싱/끝에서 인덱스 구문을 보존합니다.

### 42.2 경계 논리

직접 백엔드 슬라이싱이 가능한 경우, 단순한 슬라이스를 과도하게 범용적인 헬퍼 호출로 하강하는 것을 피합니다.

### 42.3 상수 슬라이스 폴딩

슬라이스 경계가 컴파일 타임 상수이고 대상이 폴딩 가능한 의미론을 가진 상수 문자열이나 배열 리터럴인 경우, 실용적이면 상수 폴딩을 수행합니다.

### 42.4 슬라이스에서 범위 재사용

핫 루프 내에서 슬라이스가 반복적으로 사용되는 경우, 의미론을 준수하면서 재사용 가능한 경계 계산의 호이스팅을 고려합니다.

---

## 43. 증가/감소 최적화 가이드

### 43.1 일반 스칼라

스칼라 가변 대상에 대한 `++` / `--`의 경우, 백엔드 네이티브 증가/감소 형태를 생성합니다.

### 43.2 알려진 자료구조 재작성과의 구분

수신자 타입이 알려질 때까지 일반 스칼라 `--x`와 알려진 자료구조 `--queue` 재작성을 혼동하지 마십시오.

### 43.3 복잡한 lvalue

인덱싱이나 속성 접근이 있는 복잡한 lvalue의 증가/감소에 대해, 평가 순서를 보존하고 중복된 부수 효과를 피합니다.

---

## 44. 암시적 반환 최적화 가이드

### 44.1 명시적 백엔드 `return` 선호

암시적 반환 의미론을 해석한 후, 함수 또는 람다 본문 문맥에서는 항상 명시적 백엔드 `return` 구문을 생성합니다.

### 44.2 표현식 청크 방지

블록 표현식을 백엔드 표현식 형태로 직접 매핑할 수 없는 경우, 블록 의미론을 컴파일러 생성 람다 내부에 숨기는 대신 주변 구조를 구문 형태로 재작성합니다.

### 44.3 단독 호출 규칙 준수

마지막 단독 호출이 암시적 반환이 아니라는 규칙을 보존합니다.

이것을 값 반환으로 "최적화"하지 마십시오.

---

## 45. 지역 상수 폴딩 및 단순화

의미론적 하강 후 최종 생성 전에, 경량 지역 단순화 패스를 권장합니다.

좋은 대상으로는 다음이 있습니다:

1. 리터럴에 대한 산술,
2. 불리언 단순화,
3. 상수와의 비교,
4. 상수에 대한 변환 키워드 호출,
5. 단순한 비교 트리 평탄화,
6. 불필요한(no-op) 임시 변수 제거.

이 패스는 지역적이고 보수적으로 유지해야 합니다.

---

## 46. 임시 변수 규율

생성된 임시 변수는 종종 필요하지만, 다음 원칙을 따라야 합니다:

1. 필요한 경우에만 도입하고,
2. 범위를 작게 유지하며,
3. 소스 평가 순서를 보존하고,
4. 일반적인 핫 하강 코드에서 불필요한 변수를 피하며,
5. 빌더 및 집계 하강에서 중첩된 임시 변수 폭발을 피합니다.

권장 이름 계열:

```txt
__pscpTmp
__pscpIter
__pscpAcc
__pscpDiscard
__pscpCmp
```

## 47. 최적화 제약으로서의 가독성

생성된 코드의 품질은 중요합니다.

가독성 높은 생성 C#은 디버깅이 쉽고 추가 최적화도 더 쉽습니다.

두 하강이 성능이 비슷한 경우, 다음을 만족하는 것을 선호합니다:

- 직접 루프를 보여주고,
- 소스의 가시적 구조를 보존하며,
- 깊이 중첩된 헬퍼 호출을 피하고,
- 불필요한 이터레이터 및 람다 노이즈를 피하며,
- 거대한 단일 헬퍼 추상화를 피합니다.

---

## 48. 권장 최적화 구현 순서

최적화를 구현하는 실용적인 순서는 다음과 같습니다:

1. 직접 스칼라 입력 하강,
2. 직접 고정 형태 배열 입력 하강,
3. 스칼라/튜플/1차원 컬렉션에 대한 직접 출력 렌더링,
4. 단순 범위에 대한 직접 `for` 하강,
5. 정적 범위에 대한 직접 구체화된 빌더 채우기,
6. 범위/제너레이터에 대한 집계 융합,
7. `chmin` / `chmax` 직접 재작성,
8. 알려진 자료구조 연산자 재작성,
9. `new![n]` 직접 채우기,
10. 변환 키워드 특수화,
11. 슬라이싱 보존 및 정리,
12. 비교자 캐싱 및 2차 정리.

## 49. 참조 백엔드를 위한 점검 목록

좋은 `v0.4` 트랜스파일러 백엔드라면 다음 질문에 "예"라고 답할 수 있어야 합니다.

1. 단순 숫자 케이스에서 `0..<n -> i { ... }`가 `for` 루프가 되는가?
2. `sum (0..<n -> i do f(i))`가 중간 할당을 피하는가?
3. `int[n] arr =`가 특수화된 스캐너 호출의 루프로 하강되는가?
4. `HashSet += x`가 기본 `bool` 반환값을 보존하는가?
5. `new![n]`이 할당 하나와 직접 초기화 루프 하나가 되는가?
6. C#이 `_`를 식별자로 취급할 수 있는 경우에도 `_`가 의미론적으로 진정한 버리기로 남는가?
7. 백엔드가 집계 핫 경로에서 `dynamic`을 피하는가?
8. 백엔드가 단순 범위 루프에서 헬퍼 열거자를 피하는가?
9. 일반 구문 하강에서 컴파일러 생성 람다가 피해지는가?
10. `sumBy`가 기본 핫 하강에서 `Select(...).Sum()`을 피하는가?
11. 일반적인 스칼라 케이스에서 변환 키워드 호출이 직접 하강되는가?
12. 문제가 발생했을 때 디버깅할 수 있을 만큼 생성된 코드가 여전히 가독성 높은가?

## 50. 마무리 노트

PSCP 최적화에서 가장 중요한 아이디어는 단순합니다:

- 소스 코드는 짧게 만들고,
- 생성된 코드는 직접적으로 유지하며,
- 트랜스파일러가 구조적 작업을 수행하게 하고,
- 컴파일 타임 하강으로 제거할 수 있는 문법 설탕의 비용을 런타임에 지불하지 않습니다.

이것이 이 가이드가 강제하고자 하는 기준입니다.
